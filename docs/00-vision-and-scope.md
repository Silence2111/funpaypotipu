# 00 — Видение, сегменты рынка и scope

## 1. Контекст рынка

Рынок цифровых игровых маркетплейсов Рунета — 5–6 заметных площадок, совокупный
GMV ≈ 15–25 млрд ₽/год, доход площадок ≈ 1–2 млрд ₽/год. Драйверы: огромная игровая
аудитория СНГ + платёжная изоляция РФ (спрос на посредников для оплаты Steam, доната,
зарубежных подписок).

| Площадка | Сегмент | Суть |
|----------|---------|------|
| **FunPay** | P2P-ценности | Лидер, ~29M визитов/мес, GMV 5–10 млрд ₽. Эскроу + чат + комиссия |
| **Playerok** | P2P-ценности | Быстрорастущий (2022), агрессивный маркетинг, двойная комиссия |
| **LZT (Lolz)** | P2P + «серое» | Падающий трафик, блокировки РКН |
| **GGSel / Plati** | Ключи/цифра | Площадка цифровых товаров, авто-выдача |
| **Kupikod** | Пополнения | Пополнение Steam/сервисов, комиссия 10–20%, подписка |

**Ключевые выводы для нас:**

1. **Органический трафик — это ров (moat).** У лидера десятки млн визитов из поиска.
   → SSR/SEO не опция, а требование архитектуры (см. [07](07-search-and-seo.md)).
2. **Эскроу + защита сделки — ядро доверия и удержания денег на площадке.**
   Без анти-скама пользователи уходят договариваться напрямую, минуя комиссию.
3. **Самая высокая маржа — в пополнениях и донате.** Туда сходятся все игроки.
4. **Окно входа — не лобовая конкуренция.** Telegram-нативность, B2B/инфраструктура,
   качество UX и скорость выплат — реальные дифференциаторы.

## 2. Видение продукта

> Маркетплейс, где покупатель безопасно покупает любой цифровой игровой товар, а
> продавец быстро получает деньги — с минимумом трения и максимумом защиты сделки.

Мы строим **гибрид двух моделей** на одном ядре:

- **P2P-эскроу** (ядро, как FunPay/Playerok): продавцы-люди выставляют лоты,
  площадка держит деньги до подтверждения.
- **Авто-выдача / пополнения** (расширение, как Kupikod/GGSel): товар выдаётся
  автоматически из склада ключей или через интеграцию с провайдером.

Архитектурно это **один `Order` с разными стратегиями исполнения (fulfillment)** —
см. [03](03-escrow-and-ledger.md). Это даёт охват всего рынка без раздвоения кодовой базы.

## 3. Сегменты товаров (категории верхнего уровня)

| Сегмент | Тип выдачи | Риск | Пример |
|---------|-----------|------|--------|
| Аккаунты | Ручная (передача данных) | Высокий (угон, чарджбэк) | Аккаунт Steam с играми |
| Валюта/предметы | Ручная (передача в игре) | Средний | Голда WoW, скины CS2 |
| Услуги/буст | Ручная (выполнение работы) | Средний | Прокачка до ранга |
| Ключи/цифра | **Авто** (из склада) | Низкий | Ключ игры, подписка |
| Пополнения/донат | **Авто** (через провайдера) | Низкий-средний | Пополнение Steam-кошелька |

## 4. Роли пользователей

- **Покупатель (Buyer)** — ищет, покупает, подтверждает, оставляет отзыв, открывает спор.
- **Продавец (Seller)** — выставляет лоты, общается, выдаёт товар, получает выплаты.
  (Один аккаунт может быть и тем, и другим.)
- **Арбитр / Модератор (Agent)** — разбирает споры, модерирует контент и пользователей.
- **Администратор (Admin)** — управляет платформой, комиссиями, ролями, настройками.
- **Финансовый оператор** — реконсиляция, выплаты, возвраты (отдельная роль/доступ).

## 5. Бизнес-модель (монетизация)

1. **Комиссия со сделки** — основной доход. Гибкая: % + фикс, разная по категориям,
   возможна двойная (с покупателя и продавца — модель Playerok).
2. **Платное продвижение лотов** — поднятие в выдаче, выделение.
3. **Маржа на пополнениях** — спред курса + комиссия (высокомаржинальный сегмент).
4. **Подписки для продавцов** — сниженная комиссия, расширенные лимиты, аналитика.
5. **Эквайринг-спред** — на вводе/выводе средств.

Конфигурация комиссий — данные, а не код: таблица `FeeRule` (см. [02](02-domain-model.md)).

## 6. Scope: что входит и что нет

### MVP-ядро (фазы 0–3, см. [11](11-roadmap.md))

✅ Регистрация/auth (email, OAuth, Telegram), профили, рейтинги
✅ Каталог: игры → категории → атрибуты → лоты
✅ Поиск и фильтры по атрибутам
✅ Эскроу-сделка (машина состояний) + кошелёк + двойная бухгалтерия
✅ Базовые платежи (1 провайдер) и выплаты
✅ Встроенный чат с маскированием контактов
✅ Отзывы и рейтинги
✅ Споры/арбитраж (базовый) + админ/модерация

### Расширение (фазы 4–6)

🔵 Авто-выдача ключей (склад) и пополнения через провайдеров
🔵 Антифрод-движок (скоринг, мультиаккаунты, чарджбэк-защита)
🔵 Масштабируемый поиск (Meilisearch/OpenSearch), продвинутый SEO
🔵 Мобильное приложение / Telegram Mini App
🔵 i18n, мультивалютность, B2B-API

### Вне scope (сознательно)

❌ Собственный платёжный шлюз/процессинг (используем провайдеров)
❌ Криптобиржа/обмен (возможна интеграция, не ядро)
❌ Хостинг игровых серверов, лаунчер, античит

## 7. Нефункциональные требования (цели)

| Свойство | Цель |
|----------|------|
| Доступность | 99.9% для критичных путей (платежи, сделки) |
| Согласованность денег | **Strong** — двойная запись, никаких потерянных/двойных движений |
| Идемпотентность | Все денежные операции и вебхуки идемпотентны |
| Аудируемость | Полный неизменяемый журнал денег и действий модерации |
| Производительность | Каталог/лот: TTFB < 200ms (SSR+кэш), p95 API < 300ms |
| SEO | SSR всех публичных страниц, Core Web Vitals «green» |
| Безопасность | См. [09](09-security.md): 2FA, RBAC, шифрование PII, защита выдачи товара |
| Масштаб | Горизонтальное масштабирование API/worker, read-replica БД |

## 8. Глоссарий

- **Лот (Listing/Offer)** — предложение продавца о продаже товара.
- **Сделка (Order)** — конкретная покупка лота покупателем.
- **Эскроу (Escrow)** — удержание денег площадкой до завершения сделки.
- **Выдача (Fulfillment)** — передача товара покупателю (ручная/авто).
- **Ledger** — двойная бухгалтерская книга всех движений денег.
- **Спор (Dispute)** — конфликт по сделке, разбираемый арбитром.
- **Холд выплат (Payout hold)** — задержка вывода средств продавцом (антифрод).
